  # Agent-in-the-Loop Forecasting Constitution

  ## Core Principles

  ### I. Tool-Grounded Agent (NON-NEGOTIABLE)
  The agent is a bounded junior analyst, not a forecaster. It must inspect
  diagnostics and
  metrics via tools before any recommendation; it may only choose from a
  predefined
  intervention menu; it must never invent evidence, cite uncomputed metrics, or
  use tools
  outside the allowed set. Explanations must separate evidence from speculation.

  ### II. Backtest-Validated Interventions (NON-NEGOTIABLE)
  Backtesting is the guardrail. No intervention is accepted unless a rolling
  backtest shows
  it improves the chosen metric (or improves reliability without unacceptable
  accuracy loss).
  The agent must rerun backtests after every intervention and reject its own
  recommendation
  when results worsen.

  ### III. Shared Core, Independent Pipelines
  `src/ailf/core/` is the common, stable spine (models, backtest, metrics,
  agent, eval,
  reporting). Each use-case lives in `src/ailf/pipelines/<usecase>/` and owns
  only its
  dataset generation, diagnostic tool, and prompts. Pipelines must not depend on
   each other.
  Core changes require review and must keep all core tests green.

  ### IV. Test-First Diagnostic Tools
  Every diagnostic tool (drift / changepoint / outlier) ships with unit tests
  against
  synthetic series with KNOWN injected ground truth
  (precision/recall/false-positive),
  written and passing before the tool is integrated with the agent.

  ### V. Reproducibility & Versioned Prompts
  One uv-locked environment for everyone. All synthetic data generation is
  seeded. All
  prompts (agent and LLM-judge) are versioned ‚ÄĒ a released prompt is never
  edited in place;
  a new version is added and referenced by code and reports.

  ### VI. Baseline Comparison Always
  Seasonal Naive and Prophet-default are the mandatory references. A pipeline
  result that
  cannot be compared against them is incomplete. Datasets where the issue does
  not actually
  reproduce are filtered out of the train/test set.

  ### VII. Session Transparency
  Every member exports their Claude sessions to `sessions/<person>/` for
  provenance and
  grading. Significant design decisions are recorded as short ADRs under
  `docs/decisions/`.

  ## Technology Constraints
  - Python managed by uv (single workspace). Package: `ailf`.
  - Time series & models via Darts (Prophet, AutoARIMA/ETS, Seasonal Naive);
  Prophet as the
    interpretable analyst-in-the-loop baseline. Chronos/TimesFM only "if time
  permits".
  - Agent via LangGraph/LangChain; LLM via Anthropic; eval tracing via LangSmith
   (optional).
  - Metrics: MAE, RMSE, MASE, WAPE, sMAPE, prediction-interval coverage.
  - POCs in `pocs/` are exempt from these constraints to enable fast
  experimentation.

  ## Development Workflow
  - Spec-driven: `/speckit.specify` ‚Üí `/speckit.plan` ‚Üí `/speckit.tasks` ‚Üí
  implement, one
    feature branch (`NNN-<name>`) per task, PR'd into `main`.
  - Work stays inside your pipeline's directories to avoid cross-team merge
  conflicts.
  - PRs must keep tests green; PRs touching `core/` get an extra review.
  - Fail-fast: prove a use-case in `pocs/` before investing in a full pipeline.

  ## Governance
  This constitution supersedes ad-hoc practice. The two NON-NEGOTIABLE
  principles (I and II)
  may not be bypassed for convenience. Amendments require a PR, team agreement,
  and a note
  in `docs/decisions/`. Complexity must be justified against these principles.

  # SPEC: Agent-in-the-Loop Forecasting

  ## 1. Problem & Research Question
  Automatic forecasting models fail silently on drift, changepoints, outliers,
  and regime
  changes; expert analysts catch these but don't scale. **Can a bounded LLM 
  agent improve a
  forecasting pipeline by diagnosing common failure modes, recommending 
  interventions, and
  validating them through backtesting?** The agent plays the analyst role only ‚ÄĒ
   models
  forecast, tools diagnose, the agent reasons/explains/validates. (See `docs/`
  proposal.)

  ## 2. System Architecture (5 layers)
  1. **Data** ‚ÄĒ synthetic (seeded, knob-driven via Darts) + standard datasets.
  2. **Forecasting models** ‚ÄĒ Seasonal Naive, Prophet, AutoARIMA/ETS
  (Chronos/TimesFM optional).
  3. **Diagnostics** ‚ÄĒ drift / changepoint / outlier detection tools + metric
  computation.
  4. **LLM analyst agent** ‚ÄĒ ReAct loop: reads diagnostics, picks tools,
  proposes bounded
     interventions, writes explanations.
  5. **Evaluation** ‚ÄĒ rolling backtests accept/reject interventions; LLM judge +
   manual evals.

  ## 3. Use-cases (parallel team tracks)
  | Track       | Owners                 | Deliverables
                 |
  |-------------|------------------------|--------------------------------------
  ---------------|
  | Drift       | Sowmya, Harish         | dataset gen, drift tool, prompts,
  pipeline, evals   |
  | Changepoint | Goutham, Meena, Dinesh | dataset gen, changepoint tool,
  prompts, pipeline    |
  | Anomaly     | Nadhiya, Amudhaval     | dataset gen, outlier tool, prompts,
  pipeline        |
  All three plug into the shared `core/` agent + backtest + eval spine.

  ## 4. Capability roadmap (priority order, from the proposal)
  1. Detect drift ‚Üí 2. Identify changepoints ‚Üí 3. Flag outliers ‚Üí 4. Compare
  baselines ‚Üí
  5. Recommend interventions (bounded menu) ‚Üí 6. Validate via backtesting ‚Üí 7.
  Explanation report.

  ## 5. MVP (the minimum complete loop)
  Load dataset ‚Üí fit Prophet + Seasonal Naive ‚Üí backtest ‚Üí detect issue ‚Üí agent
  diagnoses &
  picks from a small intervention menu ‚Üí refit ‚Üí compare before/after metrics ‚Üí
  write report.

  ## 6. Intervention menu (bounded)
  Add/relax changepoints; additive‚ÜĒmultiplicative seasonality; add seasonality;
  treat interval
  as outlier-contaminated; add holidays/regressors; switch baseline; retrain on
  recent window.
  Each recommendation states: symptom, likely cause, intervention, deciding
  metric.

  ## 7. Evaluation
  Diagnostic correctness (P/R/FPR on injected ground truth), tool-use
  correctness, intervention
  quality, backtest discipline, forecast outcome vs. baselines, constraint
  adherence, explanation
  quality, stability. Baselines to beat: Prophet-default, Seasonal Naive,
  rule-based loop,
  LLM-only (no tools). LLM-judge prompts are versioned.

  ## 8. Milestones (fail-fast ‚ÄĒ fill in dates)
  - M0 Foundation (this PR): structure, CLAUDE.md, SPEC.md, constitution. ‚ÄĒ TBD
  - M1 POCs: each track proves its issue reproduces + is detectable in `pocs/`.
  ‚ÄĒ TBD
  - M2 Datasets + problem reproduction (filtered train/test). ‚ÄĒ TBD
  - M3 Tools + unit tests; agent integration (MVP loop). ‚ÄĒ TBD
  - M4 Evals + LLM judge; report generation. ‚ÄĒ TBD
  - M5 (optional) extended baselines / UI / chat. ‚ÄĒ TBD

  ## 9. Out of scope (unless time permits)
  Direct numerical forecasting by the LLM; production deployment; chat UI;
  foundation-model
  baselines (Chronos/TimesFM). These are explicitly "extended version" items.
